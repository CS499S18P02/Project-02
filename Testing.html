Testing


1. Unit Testing:

  For this section of testing we will be looking at the specific web pages; the specific pieces of each page independently. This will give us any errors in each individual chunk. The type of testing we will be doing is mainly equivalence partitioning and boundary value analysis, meaning we will test whether the text field limits work. This will give us a good overview of any errors we have with inputs and the boundaries of those inputs.The list of pages we will check is...

  • Website homepage
  • Login page
  • Sign-up page
  • Registration page 1
  • Registration page 2
  • Registration page 3
  • Profile page
  • User Homepage
  • Admin groups page
  • Admin users page
  • Admin skills page
  • Admin features page

Each of the above pages will be tested in Google Chrome first and we will see what other browsers and browser versions we will be able to test on in the time constraint.

2. Integration/Function Testing:

  During this phase of testing we will be looking at the functionality of the website as a whole. We will be either using other testers or, more likely, those that didn't work on the part of the website being tested. This will reduce programming bias and help find errors. These Users will use Black-Box techniques as well as others to assess the functionality of the website. The functionality we will be looking for will be...

  • Website homepage
  • Ability to click on every button and functional part of the homepage.
  • Login page
  • Be able to input the appropriate information into the fields for input.
  • Check for input error messages
  • Check boundary cases with alpha/numeric characters and character limits.
  • Sign-up page
  • Be able to input the appropriate information into the fields for input.
  • Check for input error messages
  • Check boundary cases with alpha/numeric characters and character limits.
  • Registration page 1
  • Be able to input the appropriate information into the fields for input.
  • Check for input error messages
  • Check boundary cases with alpha/numeric characters and character limits.
  • Check radio buttons, calendar functionality, and dropdown menu.
  • Registration page 2
  • Test skills box functionality.
  • Registration page 3
  • Test skills box functionality.
  • Profile page
  • See if previosly input information is saved onto the profile page.
  • See if the save/edit functionality works.
  • Check all the same fields as the registration including boundary cases.
  • User Homepage
  • See if the phone sign-up fields work. i.e. check boundary cases as well as input types.
  • See if the buttons are functional.
  • Admin groups page
  • Here we have many buttons to test. Try the functionality of each one and make sure that it does what the button is supposed to correspond to.
  • See if the menus that pop up when selected work like the user would think they would.
  • Admin users page
  • Make sure you see accurate information about all users.
  • check whether the admin can delete and edit users that are below their promotion level.
  • Make sure users can see what they are allowed to see and edit what they should be allowed to edit on this page.
  • Test whether you can export the users information onto an excel file.
  • Admin skills page
  • See if the admin can create a new skill for the skills page and delete skills from the list.
  • Check whether these changes actually occur for other users.
  • Admin features page
  • Test whether you can add features to this page.
  • Test whether you can download features.
  • Check whether you can add features to specific groups of users.

3.Systems Testing:

  We would like to do extensive testing in this area however with the time constraint we will be first testing in Google Chrome and we may get to test in other browsers/environments. This is mainly because the customer didn't specify specific environment functionalities with this website. In Google Chrome we will be mainly testing the usability of the website first and go from there.

4.Customer/User Testing:

  We have been in contact with the customer throughout the development of this project and they have looked at the functionality of the project throughout the semester. They have been very good about telling us when they want a functionality included or if one of their required functions didn't work for them. For this part of the testing plan we will be asking the customer to brief over the website to see if they can find any errors we missed or any functionality problems. Since this is a Quality Assurance company we can be sure that they will give useful feedback as they have throughout the semester.
